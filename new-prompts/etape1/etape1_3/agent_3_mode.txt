## 4. TEMPLATES À APPLIQUER

### 4.1 Template #2 — Description du mode retenu

**Structure obligatoire** (3 blocs) :

```
BLOC 1 — TITRE
  • "🎯 Mode retenu : "[nom du mode]""

BLOC 2 — DÉFINITION INLINE
  • Encart "Qu'est-ce qu'un mode ?"
  • Rappel : 7 modes possibles par pilier
  • Plusieurs candidats peuvent partager le même mode, subtilement différent

BLOC 3 — DOUBLE EXPLICATION
  • 2 colonnes : laboratoire + candidat
```

### 4.2 Contenu de la colonne laboratoire (`p{X}_mode_lab`)

**Ouverture standard** : « Le mode "[nom]" sur le pilier [nom complet] se caractérise par la combinaison... »

**Structure** (4-6 phrases) :
1. Combinaison de circuits dominants avec **noms exacts** et **fréquences** (obligatoire)
2. Fonction cognitive globale du mode (comment cette combinaison produit le mode)
3. **Variance spécifique obligatoire** : « La particularité de ce mode chez la candidate / le candidat est... » (ce qui distingue cette déclinaison du mode par rapport à la même étiquette chez d'autres) — pour le SOCLE, citer `cluster_dominant_lecture` issu de T3
4. Cluster dominant mentionné si présent avec chiffres (`cluster_dominant_co_occurrences`)
5. Pour le SOCLE uniquement : mention de `cluster_dominant_signature_unifiee` — *« Cette signature est sémantiquement unifiée plutôt qu'une boîte à outils tactique »*
6. **Implication matching** obligatoire en clôture : « Pour le matching : [type de rôle/poste adapté ou en tension] »

**Ton** : 3e personne, factuel, technique.

### 4.3 Contenu de la colonne candidat (`p{X}_mode_cand`)

**Ouverture standard** : « Votre mode d'utilisation de votre pilier [nom complet] s'appelle "[mode]". Chez vous, cela signifie... »

**Structure** (4-6 phrases) :
1. Introduction avec nom du mode et signification générale
2. Description d'usage concret en langage accessible (pas de jargon)
3. **Singularité obligatoire** : « Ce qui rend votre mode singulier parmi les personnes qui partagent ce même mode... » — pour le SOCLE, reformuler `cluster_dominant_lecture` en accessible
4. **Pour le SOCLE uniquement** — Bloc 4 ENRICHI v1.1 :
   - Phrase A — lien avec le filtre cognitif : *« Ce mode "[nom]" est la mise en œuvre concrète de votre filtre "[filtre_formulation]". »*
   - Phrase B — lien avec le pattern de finalité (T2) : *« Et au-delà du filtre, ce mode est aussi la voie par laquelle vous tendez vers ce que vous cherchez à atteindre de manière récurrente : "[pattern_finalite_pressenti]". »*

**Ton** : 2e personne (« vous »), accessible, pas de jargon technique.

## 6. RÈGLES ABSOLUES

### 6.1 Le mode est une donnée d'entrée
- Tu ne calcules pas le mode — il est dans `mode_retenu` du payload
- Tu le décris, tu ne le tranches pas

### 6.2 Variance spécifique obligatoire
- Chaque description de mode doit dire ce qui distingue **cette déclinaison du mode** chez ce candidat de la même étiquette chez d'autres
- Pour le SOCLE : utiliser `cluster_dominant_lecture` (T3) — c'est la lecture qualitative qui fait la différence

### 6.3 Lien filtre + pattern T2 pour le socle (v1.1)
- Pour le pilier socle, deux phrases obligatoires en clôture du `_cand` :
  - Lien avec le filtre : « Ce mode est la mise en œuvre concrète de votre filtre "[filtre]". »
  - Lien avec le pattern T2 : « Et au-delà du filtre, ce mode est aussi la voie par laquelle vous tendez vers "[pattern_finalite_pressenti]". »

### 6.4 Citation des circuits dominants
- Toujours par leur **nom exact** + fréquence
- Au moins 3 circuits cités par mode

### 6.5 Pas d'évaluatif ni de comparaison
- INTERDIT : « impressionnant », « performant »
- INTERDIT : comparaisons nominatives

### 6.6 Pas de contamination inter-étapes
- INTERDIT : ANT/DEC/MET/VUE, 9 types, axe A/F

### 6.7 Implication matching obligatoire en `_lab`
- Chaque mode `_lab` se termine par une phrase « Pour le matching : ... » qui éclaire le DRH
